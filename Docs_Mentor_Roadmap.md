## Общая цель

«« [[Docs_Mentor_Project]]

Создать платформу для работы с технической документацией на основе RAG-архитектуры, позволяющую пользователю создавать «библиотеки знаний» из документации фреймворков/библиотек и вести с ними чат с использованием LLM.

## Блок 1. Фундамент (Django: пользователи, библиотеки, API)

»» [[Block_1_Django]]
### Задачи

- Установка и настройка Django + Django REST Framework
    
- Кастомная модель пользователя
    
- Регистрация / авторизация (JWT или сессии — на твой выбор)
    
- OAuth2.0 (Google / Yandex) — опционально, если успеваем
    
- Модели:
    
    - `Library` (название, ссылка, статус парсинга, дата создания, владелец)
        
    - `Chat` (название, библиотека, дата создания, настройки: модель, промт, top_k)
        
    - `Message` (чат, роль, текст, дата)
        
- CRUD API для библиотек и чатов
    
- Базовый фронт (можно просто Swagger/DRF browsable API для начала)
    

### Технологии

- Django, DRF
    
- PostgreSQL
    
- Redis (для кешей и Celery брокера — подготовка)
    

### Результат к концу блока

- Пользователь может зарегистрироваться, создать библиотеку (по ссылке), создать чат, отправлять сообщения (сохраняются в БД, но без LLM и RAG).
    

### Не делаем в этом блоке

- Парсинг
    
- LLM
    
- RAG
    
- FastAPI

## Блок 2. Парсинг, эмбеддинги, векторная БД

### Задачи

- Celery + Redis: фоновая задача парсинга документации
    
- Абстрактный класс `BaseParser` + адаптеры:
    
    - MkDocs (FastAPI)
        
    - Sphinx (PyTorch, Python docs) — опционально на старте
        
- Разбивка страниц на чанки (правила: 500-1000 токенов, overlap 10-20%)
    
- Генерация эмбеддингов (через OpenAI `text-embedding-3-small` или локальную модель)
    
- Сохранение в PGVector (таблица `document_chunks`: `library_id`, `chunk_text`, `embedding`, `source_url`, `metadata`)
    
- Статусы библиотеки: `pending` → `parsing` → `ready` / `failed`
    
- Эндпоинт проверки статуса библиотеки
    

### Технологии

- Celery, Redis
    
- PGVector (расширение PostgreSQL)
    
- OpenAI API (эмбеддинги) — пока через пользовательский ключ
    
- BeautifulSoup / Scrapy (по необходимости)
    

### Результат к концу блока

- Пользователь добавляет ссылку → через N минут библиотека готова → можно выполнять поиск по документации (без LLM, просто вывод релевантных чанков через API).
    

### Не делаем в этом блоке

- Чат с LLM
    
- FastAPI
    
- Кеширование блоков чата

## Блок 3. RAG-чат (FastAPI, LLM, стриминг)

### Задачи

- FastAPI-сервис:
    
    - Эндпоинт `POST /chat/{chat_id}/message` — стриминг ответа
        
    - RAG-логика:
        
        1. Принять запрос пользователя
            
        2. Поиск top_k чанков в PGVector (по библиотеке чата)
            
        3. Загрузить контекст чата (последние N сообщений + опциональное саммари)
            
        4. Сформировать промт
            
        5. Вызвать LLM (OpenAI, ключ пользователя)
            
        6. Стримить ответ
            
        7. Сохранить сообщение (через API Django, либо напрямую в БД — обсудим)
            
- Саммаризация контекста (опционально, по порогу токенов):
    
    - Фоновая задача в Celery
        
    - Замена старых сообщений на краткое саммари
        
- Кеширование релевантных блоков для чата (Redis) — упрощённая версия без сложной логики (сброс через 30 минут или при смене темы)
    
- Эндпоинт для смены модели / промта внутри чата
    

### Технологии

- FastAPI (отдельный процесс)
    
- Redis (кеш блоков, брокер)
    
- Celery (для саммаризации)
    
- OpenAI API (LLM, эмбеддинги уже сделаны в блоке 2)
    

### Результат к концу блока

- Полноценный RAG-чат: пользователь задаёт вопрос, получает ответ с цитатами из документации, контекст чата поддерживается.
    

### Не делаем в этом блоке

- Перевод запросов/ответов
    
- Альтернативные LLM-провайдеры
    
- Сложную агентскую логику (LangGraph)

## Блок 4. Полировка, опции, UI, безопасность

### Задачи

- Фронтенд (минимальный, но удобный):
    
    - Список библиотек пользователя
        
    - Список чатов внутри библиотеки
        
    - Окно чата с стримингом (SSE или WebSockets)
        
    - Настройки чата (модель, промт, top_k, включение саммаризации)
        
- Функции:
    
    - Перевод запроса/ответа (опционально, через маленькую модель `Helsinki-NLP/opus-mt`)
        
    - Кнопка «обновить библиотеку» (перепарсить документацию)
        
    - Добавление дополнительной документации к существующей библиотеке (новые URL)
        
- Безопасность:
    
    - Разделение system / user промтов в API
        
    - Экранирование пользовательского ввода
        
    - Проверка API-ключа пользователя перед вызовами OpenAI
        
- Документация к проекту (README, архитектура, инструкция по запуску)
    

### Технологии

- React / простой HTML+JS + EventSource (для стриминга)
    
- Nginx (прокси)
    
- Docker Compose (для простого развёртывания)
    

### Результат к концу блока

- Полностью рабочий продукт, готовый к демонстрации. Пользователь заходит, создаёт библиотеку, чатится с документацией, переводит ответы, управляет настройками.
